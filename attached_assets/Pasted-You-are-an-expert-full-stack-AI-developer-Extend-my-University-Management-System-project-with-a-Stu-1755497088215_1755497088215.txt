You are an expert full-stack AI developer. Extend my University Management System project with a Student–Faculty Request Callback feature.

Requirements
Roles

Student and Faculty accounts already exist with separate dashboards and databases.

Only Students can create callback requests.

Faculty can only view requests assigned to them and accept/reject/mark them as completed.

Feature Flow

Student Flow

Student logs in → Goes to Request Callback section.

Fills form:

Select Faculty (dropdown from faculty table)

Subject/Reason (text input)

Preferred Date & Time (datetime input)

Request saved in DB with status = Pending.

Student can view all requests in My Requests with statuses (Pending / Accepted / Rejected / Completed).

Faculty Flow

Faculty logs in → Sees Requests to Me section.

Each request shows: Student Name, Roll No, Subject, Preferred Time, Current Status.

Faculty can Accept / Reject / Mark Completed.

Status change should immediately reflect in both Student and Faculty dashboards.

Database Schema

Add new table:

CREATE TABLE callback_requests (
    id INT AUTO_INCREMENT PRIMARY KEY,
    student_id INT,
    faculty_id INT,
    subject VARCHAR(255),
    preferred_time DATETIME,
    status ENUM('Pending','Accepted','Rejected','Completed') DEFAULT 'Pending',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (student_id) REFERENCES students(id),
    FOREIGN KEY (faculty_id) REFERENCES faculty(id)
);

Backend Requirements

API endpoints (Node.js/Express or Django/Flask if Python):

POST /student/request-callback → Student creates request

GET /student/:id/requests → Fetch student’s requests

GET /faculty/:id/requests → Fetch faculty’s requests

PUT /faculty/request/:id → Faculty updates request status

Enforce role-based access control.

Only students can create requests.

Only assigned faculty can update requests.

Student Dashboard

"My Requests" table showing Faculty, Subject, Preferred Time, Status.

Faculty Dashboard

"Requests to Me" table with Student details + Status.

Buttons for Accept / Reject / Mark Completed.

UI framework: React + Tailwind (or HTML/CSS if simpler).

Deliverables

Database migration script.

Backend API code with authentication & validation.

Frontend components (React + Tailwind preferred).
